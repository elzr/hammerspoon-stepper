Confirmed and diagnosed. This isn't Hammerspoon or your config — it's a macOS-level block, and I can prove it.

## Root cause: Secure Input is stuck ON, held by Google Chrome

`ioreg` reports `kCGSSessionSecureInputPID=983`, and **PID 983 is Google Chrome** (running since yesterday 09:31). When an app calls `EnableSecureEventInput` (Chrome does this for password fields) and never releases it, macOS suppresses keyboard delivery to **every CGEventTap system-wide** — the built-in anti-keylogger mechanism. A logout/reboot clears it, which is exactly why you sensed "a full reload would fix it."

## Why both systems died together — a hidden shared dependency

The common factor isn't Hyperkey. It's *"anything built on an event tap"*:

- **Hyperkey** is a CGEventTap (intercepts Caps Lock → synthesizes ⌘⌃⌥⇧). Blocked → Caps Lock stops producing the hyper modifier → hyper-based stepper hotkeys silently fail. Hyperkey is a *victim* here, not the cause.
- **rcmd** is also a CGEventTap (watches right-cmd/ropt + letter). Same block → `ropt+g` and friends dead.

## The fingerprint that proves it

Hammerspoon itself is **healthy** — IPC responds, all **58 hotkeys still registered**, and the console shows a *plain* stepper hotkey firing 2 min ago:

```
08:43:16  trigger: move-to-display:left 'kitty'
```

That's `ctrl+alt+←` (`stepper.lua:1022`). Plain `hs.hotkey.bind` uses Carbon `RegisterEventHotKey`, which secure input does **not** suppress — so those keep working while the event-tap layer (Hyperkey, rcmd) is blacked out. That selective pattern *is* the signature of stuck secure input.

**Quick test for you right now:** `ctrl+alt+arrow` window moves should still work; Caps-Lock-hyper combos and rcmd toggles won't.

## The fix (no full reboot)

Quitting & relaunching **Chrome** releases the lock the instant the process exits — Hyperkey/rcmd taps resume immediately. Chrome restores your tabs and won't re-grab it unless a password field gets focused again.

Want me to quit & relaunch Chrome for you (I'll verify the secure-input PID clears to `0` afterward)? Or if you'd rather not disturb Chrome's session, I can hunt down the specific tab holding the focused login field so you close just that one.
