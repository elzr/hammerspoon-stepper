OK. Let me stop flailing and do a proper code review of the cleanup logic in `flashFocusHighlight`.

The critical code (focus.lua:319-348):

```lua
function M.flashFocusHighlight(win, dir, opts)
  safeDeleteCanvas(focusHighlight)
  focusHighlight = nil
  focusHighlightGen = focusHighlightGen + 1
  local thisGen = focusHighlightGen

  local frame = win:frame()
  focusHighlight = M.createBorderCanvas(frame, dir, win, opts)

  hs.timer.doAfter(0.3, function() ... end)   -- NOT STORED
  hs.timer.doAfter(2.0, function() ... end)   -- NOT STORED
end
```

**The bug: both `hs.timer.doAfter` return values are discarded.** In Hammerspoon, if you don't store the timer reference, it's eligible for garbage collection, and a GC'd timer never fires its callback.

The sequence:
1. `flashFocusHighlight` creates canvas, starts two unstored timers
2. Lua GC runs (unpredictably) and collects both timer objects
3. Neither cleanup callback ever fires
4. `focusHighlight` still holds the canvas reference, so the canvas itself can't be GC'd either
5. The border stays on screen indefinitely — until the *next* focus navigation (which deletes the old canvas at line 321) or a Hammerspoon reload

This explains why:
- The border was stale (timers were GC'd, no cleanup fired)
- It eventually cleared during debugging (one of my IPC calls may have triggered a focus operation or the module-reload at line 13 cleared it)
- `collectgarbage` alone couldn't fix it (the canvas is still referenced by `focusHighlight`)
