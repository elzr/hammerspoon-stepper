## Claude:
The issue is the chained command - when `hs.reload()` drops the connection, the first `hs` command hangs waiting for a response it'll never get. The `2>/dev/null` only redirects stderr, but the process itself is stuck. Better approach would be to background just the reload:

```bash
/path/to/hs -c "hs.reload()" &>/dev/null & sleep 0.1; /path/to/hs -c "return 'ready'"
```

But let's commit first:

That's instant now. The `&` backgrounds the reload so it doesn't block waiting for a response that'll never come.

Going forward I'll use this pattern for reloads:
```bash
hs -c "hs.reload()" &>/dev/null & sleep 0.2; hs -c "return 'ready'"
```
